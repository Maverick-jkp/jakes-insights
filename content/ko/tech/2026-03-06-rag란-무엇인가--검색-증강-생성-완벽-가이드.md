LLM에 "어제 발표된 거 알아?" 하고 물어보면, 돌아오는 답은 항상 똑같아요. "제 학습 데이터에는 없는 내용이에요." RAG(Retrieval-Augmented Generation)는 바로 이 문제를 푸는 기술이에요. 2026년 현재, 기업 AI 도입의 절반 이상이 RAG 아키텍처를 기반으로 해요. 실제 프로덕션에서 가장 많이 쓰이는 AI 패턴이 됐죠.

이 글을 읽고 나면:
- RAG의 구조와 작동 원리를 이해할 수 있어요
- Fine-tuning, 프롬프트 엔지니어링과의 차이를 명확히 알 수 있어요
- Python으로 기본 RAG 파이프라인을 직접 만들 수 있어요
- 프로덕션 배포 전 체크해야 할 포인트를 파악할 수 있어요

---

> **Key Takeaways**
> - RAG는 LLM이 답변을 생성하기 전에 외부 지식 베이스에서 관련 문서를 검색해 컨텍스트로 제공하는 아키텍처예요.
> - Fine-tuning 대비 RAG는 학습 비용이 거의 없고, 지식 업데이트가 실시간으로 반영되어 변화가 잦은 도메인에 적합해요.
> - LangChain 0.2+와 ChromaDB를 조합하면 로컬 환경에서 30분 내에 작동하는 RAG 파이프라인을 만들 수 있어요.
> - Hallucination(환각) 감소가 RAG의 핵심 효과이며, 검색 품질이 전체 시스템 성능을 좌우해요.
> - 청킹(Chunking) 전략과 임베딩 모델 선택이 검색 정확도에 직접 영향을 미쳐요.

## RAG가 지금 이렇게 중요한 이유

GPT류 LLM은 학습 데이터 기준 이후의 정보를 몰라요. 사내 문서나 특정 도메인 지식은 아예 없고요. 이걸 해결하는 방법은 크게 세 가지였어요: 프롬프트에 문서 통째로 넣기, Fine-tuning, 그리고 RAG.

RAG의 아이디어는 2020년 Facebook AI Research(현 Meta AI)의 논문 "Retrieval-Augmented Generation for Knowledge-Intensive NLP Tasks"에서 처음 제안됐어요. 핵심은 단순해요. **검색 먼저, 생성 나중.**

사용자 질문이 들어오면:
1. 질문을 벡터로 변환
2. 벡터 DB에서 유사한 문서 검색
3. 검색된 문서를 컨텍스트로 LLM에 전달
4. LLM이 근거 있는 답변 생성

결과적으로 LLM은 "아는 척"하는 대신 실제 문서를 보고 답해요. Hallucination이 줄고, 출처를 제시할 수 있어요.

---

## RAG vs Fine-tuning vs 프롬프트 엔지니어링 비교

| 항목 | RAG | Fine-tuning | 프롬프트 엔지니어링 |
|------|-----|-------------|------------------|
| 비용 | 중간 (인프라) | 높음 (GPU 학습 비용) | 낮음 |
| 지식 업데이트 | 실시간 가능 | 재학습 필요 | 수동 업데이트 |
| 구현 난이도 | 중간 | 높음 | 낮음 |
| Hallucination 감소 | 높음 | 중간 | 낮음 |
| 확장성 | 높음 (문서 추가만) | 낮음 | 중간 |
| 도메인 특화 | 높음 | 매우 높음 | 낮음 |
| 출처 추적 | 가능 | 어려움 | 불가 |

**언제 뭘 써야 할까요?**

RAG는 내부 문서, 최신 정보, 자주 바뀌는 데이터가 있을 때 강해요. Fine-tuning은 특정 말투나 스타일, 도메인 전문 용어가 중요할 때 써요. 실제로 둘을 합치는 방식도 많이 쓰고요. 프롬프트 엔지니어링은 빠른 프로토타이핑엔 좋지만, 컨텍스트 창 한계 때문에 대용량 문서엔 한계가 있어요.

## 단계별 RAG 파이프라인 만들기

### Prerequisites

- Python 3.10+
- OpenAI API 키 (또는 로컬 모델: Ollama)
- 기본 Python 문법과 pip 사용법

```bash
pip install langchain==0.2.16 chromadb==0.5.3 openai tiktoken pypdf
```

---

### Step 1: 문서 로드와 청킹

RAG의 첫 단계는 문서를 LLM이 처리하기 좋은 크기로 자르는 거예요. 너무 크면 노이즈가 많고, 너무 작으면 맥락이 사라져요. 실무에선 보통 512~1024 토큰이 잘 맞아요.

```python
from langchain.document_loaders import PyPDFLoader
from langchain.text_splitter import RecursiveCharacterTextSplitter

# PDF 문서 로드 (회사 내부 문서, 기술 명세서 등)
loader = PyPDFLoader("company_handbook.pdf")
documents = loader.load()

# 재귀적 청킹: 문단 → 문장 → 단어 순서로 자름
# chunk_overlap은 청크 간 맥락 연속성을 위해 반드시 설정
text_splitter = RecursiveCharacterTextSplitter(
    chunk_size=800,        # 토큰 기준 약 600자
    chunk_overlap=100,     # 앞뒤 청크와 100자 겹침 (맥락 유지)
    length_function=len,
)

chunks = text_splitter.split_documents(documents)
print(f"총 {len(chunks)}개 청크 생성 완료")
```

---

### Step 2: 임베딩 생성과 벡터 DB 저장

청킹한 문서를 숫자 벡터로 바꿔서 ChromaDB에 저장해요. 이 벡터가 검색의 핵심이에요.

```python
from langchain.vectorstores import Chroma
from langchain.embeddings import OpenAIEmbeddings

# OpenAI text-embedding-3-small: 비용 대비 성능 좋음
# 로컬 환경이면 sentence-transformers 사용 가능
embeddings = OpenAIEmbeddings(model="text-embedding-3-small")

# ChromaDB에 임베딩 저장 (persist_directory로 디스크 저장)
vectorstore = Chroma.from_documents(
    documents=chunks,
    embedding=embeddings,
    persist_directory="./chroma_db"  # 재시작 후에도 데이터 유지
)

print("벡터 DB 저장 완료")
```

---

### Step 3: 검색기(Retriever) 설정

```python
# k=4: 가장 유사한 문서 4개를 검색
# MMR(Maximal Marginal Relevance) 알고리즘으로 다양성 확보
retriever = vectorstore.as_retriever(
    search_type="mmr",         # 중복 제거 + 관련성 균형
    search_kwargs={"k": 4, "fetch_k": 10}
)
```

---

### Step 4: RAG 체인 완성

```python
from langchain.chat_models import ChatOpenAI
from langchain.chains import RetrievalQA
from langchain.prompts import PromptTemplate

# 프롬프트 템플릿: 검색된 문서를 context로 전달
prompt_template = """아래 문서를 참고해서 질문에 답하세요.
문서에 없는 내용은 "문서에서 확인할 수 없어요"라고 말하세요.

문서:
{context}

질문: {question}
답변:"""

PROMPT = PromptTemplate(
    template=prompt_template,
    input_variables=["context", "question"]
)

llm = ChatOpenAI(model_name="gpt-4o", temperature=0)

# RAG 체인 조립
qa_chain = RetrievalQA.from_chain_type(
    llm=llm,
    chain_type="stuff",           # 검색 문서를 한 번에 LLM에 전달
    retriever=retriever,
    chain_type_kwargs={"prompt": PROMPT},
    return_source_documents=True  # 출처 문서도 반환
)

# 실제 질문 실행
result = qa_chain({"query": "연차는 몇 일인가요?"})
print(result["result"])
print("\n[출처]:", result["source_documents"][0].metadata)
```

## 실전 예시: 고객 지원 봇

기본 파이프라인을 응용해서 FAQ 문서 기반 고객 지원 봇을 만들어요. 실제 서비스에서 가장 많이 쓰이는 패턴이에요.

```python
import os
from langchain.document_loaders import DirectoryLoader
from langchain.vectorstores import Chroma
from langchain.embeddings import OpenAIEmbeddings

def build_support_bot(docs_directory: str):
    """
    지정된 디렉토리의 모든 .txt, .md 파일을 로드해서
    RAG 기반 고객 지원 봇을 만드는 함수
    """
    # 여러 파일 한 번에 로드
    loader = DirectoryLoader(
        docs_directory,
        glob="**/*.md",  # 마크다운 문서만
        show_progress=True
    )
    docs = loader.load()

    if not docs:
        raise ValueError(f"{docs_directory}에 문서가 없어요")

    from langchain.text_splitter import RecursiveCharacterTextSplitter
    splitter = RecursiveCharacterTextSplitter(
        chunk_size=600,
        chunk_overlap=60
    )
    chunks = splitter.split_documents(docs)

    # 벡터 DB 생성
    embeddings = OpenAIEmbeddings(model="text-embedding-3-small")
    vectorstore = Chroma.from_documents(chunks, embeddings)

    return vectorstore.as_retriever(search_kwargs={"k": 3})

# 사용 예
retriever = build_support_bot("./faq_docs")
print(f"봇 준비 완료 - 검색기 타입: {type(retriever).__name__}")
```

## 자주 하는 실수와 피하는 방법

RAG가 항상 답은 아니에요. 잘못된 설정 하나가 전체 품질을 망치거든요.

### 흔한 실수들

- **청크 사이즈를 너무 작게 설정**: 512자 미만이면 맥락이 잘려 검색 품질이 떨어져요. 800~1000자가 시작점으로 좋아요.
  - 해결: 실제 문서로 여러 크기 실험 후 정하세요

- **chunk_overlap을 0으로 설정**: 청크 경계에 걸린 정보가 사라져요. chunk_size의 10~15%로 설정하세요.
  - 해결: `chunk_overlap=chunk_size * 0.1`

- **임베딩 모델과 검색 모델 불일치**: 저장할 때와 검색할 때 다른 임베딩 모델을 쓰면 결과가 완전히 틀려요.
  - 해결: 모델 이름을 환경 변수로 통일 관리

### 프로덕션 체크리스트

- [ ] 벡터 DB 백업 및 복구 전략 수립
- [ ] 임베딩 API 비용 모니터링 설정 (토큰 사용량 추적)
- [ ] 검색 실패 시 fallback 응답 처리
- [ ] 문서 업데이트 시 재인덱싱 자동화
- [ ] 응답 품질 평가 지표(RAGAS 등) 설정
- [ ] 민감 정보 필터링 (PII 제거 후 벡터화)
