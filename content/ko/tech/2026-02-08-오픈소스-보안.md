2025년 말, 한 금융권 시스템이 3일 동안 마비됐어요. 원인은 널리 쓰이던 오픈소스 라이브러리의 숨겨진 취약점. 이 사건 이후 오픈소스 보안이 다시 뜨거운 감자가 됐죠.

지금 이 글을 읽는 순간에도 여러분의 서비스는 평균 200개 이상의 오픈소스 컴포넌트를 쓰고 있을 거예요. 문제는 그중 몇 개에 보안 구멍이 있는지 정확히 아는 팀이 거의 없다는 점이에요. GitHub의 2026년 1월 보고서에 따르면, 전체 오픈소스 프로젝트의 68%가 최소 하나 이상의 알려진 취약점을 포함하고 있더라고요.

왜 이게 중요할까요? 오픈소스는 이제 선택이 아니라 필수예요. 하지만 누구나 코드를 볼 수 있다는 건 해커도 볼 수 있다는 뜻이죠.

## 어떻게 여기까지 왔을까

2021년 말 Log4j 취약점 사태가 터졌을 때만 해도 많은 기업이 "우리는 괜찮겠지"라고 생각했어요. 그런데 2023년 이후 공격 패턴이 정교해지면서 분위기가 바뀌었죠. 해커들이 이제는 유명한 라이브러리가 아니라 의존성 체인 깊숙이 숨어 있는 작은 패키지를 노리기 시작한 거예요.

현재 상황을 보면 세 가지 변화가 눈에 띄어요. 첫째, 소프트웨어 자재명세서(SBOM) 작성이 권장에서 필수로 바뀌고 있어요. 국내 금융권과 공공기관은 2026년 상반기부터 SBOM 제출을 의무화하는 방향이에요. 둘째, 오픈소스 메인테이너들의 부담이 커졌어요. 보안 패치를 빨리 내놓지 않으면 법적 책임 문제까지 거론되니까요. 셋째, 기업들이 오픈소스를 쓰면서도 "누가 책임질 건가"라는 질문에 명확한 답을 찾지 못하고 있어요.

Snyk의 2026년 2월 통계를 보면, 평균적인 애플리케이션이 의존하는 오픈소스 패키지 수가 2023년 대비 37% 증가했어요. 복잡도가 올라갈수록 보안 관리는 더 어려워지는 구조죠.

## 진짜 문제 세 가지

### 1. 보이지 않는 의존성

오픈소스 보안에서 가장 무서운 건 "내가 뭘 쓰는지 모른다"는 점이에요.

직접 설치한 라이브러리 10개 뒤에는 그들이 의존하는 수백 개의 패키지가 숨어 있어요. npm 생태계를 예로 들면, 인기 있는 패키지 하나가 평균 80개의 다른 패키지를 끌어와요.

실제 사례를 볼까요? 2025년 10월, 한국의 한 이커머스 플랫폼이 보안 점검 중에 발견한 건데요. 결제 모듈에 쓰인 패키지 안에 4단계 깊이로 숨어 있던 이미지 처리 라이브러리에 원격 코드 실행 취약점이 있었어요. 개발팀은 그 라이브러리가 존재하는지도 몰랐죠.

Sonatype의 2026년 조사에 따르면, 기업 개발자의 73%가 자신의 프로젝트에 포함된 전체 오픈소스 컴포넌트 목록을 정확히 모르고 있다고 답했어요.

### 2. 메인테이너 한 사람이 전 세계 인프라를 지탱한다

오픈소스의 아이러니가 여기 있어요. 수백만 명이 쓰는 중요한 라이브러리가 사실은 한두 명의 자원봉사자가 퇴근 후 관리하는 거예요.

2024년 3월, xz Utils라는 압축 라이브러리에서 백도어가 발견됐어요. 이 라이브러리는 거의 모든 리눅스 배포판에 들어가는데, 메인 메인테이너는 1명이었죠. 공격자는 몇 년에 걸쳐 신뢰를 쌓은 뒤 커밋 권한을 얻어서 악성 코드를 심었어요. 다행히 마이크로소프트 엔지니어가 SSH 접속이 평소보다 500밀리초 느려진 걸 이상하게 여겨서 발견했지만요.

오픈소스 보안 재단(OpenSSF)의 데이터를 보면, 상위 500개 오픈소스 프로젝트 중 42%가 메인 메인테이너 3명 이하예요. GitHub Sponsors 통계를 보면, 월 100달러 이상 후원받는 프로젝트는 전체의 8%도 안 돼요.

이런 구조에서 보안은 선의에 의존할 수밖에 없어요. 메인테이너가 지치거나, 관심을 잃거나, 악의적인 기여자를 걸러내지 못하면 문제가 생기는 거죠.

### 3. 라이선스 컴플라이언스와 보안의 충돌

GPL이나 AGPL 같은 카피레프트 라이선스는 코드 공개 의무가 있어요. 보안상 민감한 부분까지 공개해야 할 수도 있죠. 반대로 보안을 위해 코드를 수정했는데 라이선스 의무를 안 지키면 법적 리스크가 생겨요.

2025년 하반기, 국내 한 스타트업이 이 문제로 곤란을 겪었어요. 보안 취약점을 발견해서 자체적으로 패치했는데, 해당 라이브러리가 AGPL이었던 거예요. 패치 코드를 공개하면 자사 비즈니스 로직이 드러나고, 안 공개하면 라이선스 위반이 되는 딜레마에 빠졌죠.

## 당장 무엇을 해야 할까

### 개발자라면

새 패키지 추가할 때는 몇 가지 질문을 습관화하세요. 마지막 커밋이 언제인가? 메인테이너가 몇 명인가? 알려진 취약점은 없나? npm audit, pip-audit 같은 명령어를 CI/CD에 넣는 것만으로도 기본은 챙길 수 있어요.

### 기업이라면

오픈소스 보안 정책이 없다면 지금 만들어야 해요. "우리는 어떤 라이선스까지 허용하나?", "취약점 발견 시 누가 대응하나?", "얼마나 빨리 패치해야 하나?" 같은 기준이 명확해야 혼란이 없어요. SBOM은 이제 선택이 아니라 필수예요.

### 단계별로 접근하기

**당장 1-3개월 안에:**
- 현재 프로젝트의 전체 의존성 목록 뽑아보기
- 알려진 취약점 스캔 돌려보기 (무료 도구로도 가능)
- 치명적 등급 취약점부터 우선 패치
- 개발팀 위클리 미팅에 보안 이슈 10분 슬롯 추가

**6-12개월 장기 전략:**
- SBOM 자동 생성 프로세스 구축
- 의존성 업데이트 주기 정책 수립
- 보안 챔피언 지정 (팀별로 한 명씩 교육받은 담당자)

### 이런 경우엔 조심하세요

보안 도구 잘못 고르면 개발자들이 하루 종일 false positive 처리하느라 시간 낭비해요. Gartner 2025년 조사에서 개발팀의 34%가 "보안 도구 때문에 배포가 늦어진다"고 불만을 토로했어요. 도구 도입 전에 2주 정도 파일럿을 돌려보세요. 실제 코드베이스에서 오탐률이 어느 정도인지 확인하고요.

## 앞으로 6개월

2026년 상반기에는 SBOM 의무화가 더 많은 산업으로 확대될 거예요. 금융권은 이미 시작됐고, 의료와 에너지 분야도 곧 따라올 거고요.

주목해야 할 건 AI 기반 취약점 탐지 도구의 발전이에요. 2026년 하반기쯤엔 코드 패턴만으로 아직 CVE 등록도 안 된 잠재적 취약점을 미리 찾아내는 도구들이 나올 거예요. 물론 오탐 문제는 여전할 테지만요.

다음 주 월요일 아침에 한 가지만 하세요. 여러분 프로젝트의 전체 의존성 목록을 뽑아보는 거예요. "내가 뭘 쓰고 있는지" 아는 것부터 시작이거든요. 완벽한 보안은 없지만, 더 나은 보안은 언제나 가능하니까요.
