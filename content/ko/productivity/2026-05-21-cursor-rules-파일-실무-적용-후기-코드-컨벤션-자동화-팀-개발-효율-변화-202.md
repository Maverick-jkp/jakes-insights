코드 리뷰에서 같은 말이 반복된 적 있죠? "여기 세미콜론 빠졌어요", "네이밍 컨벤션 맞춰주세요." W컨셉 프론트엔드 팀은 `.cursorrules` 파일 하나로 이 반복을 끊었어요. 2025년 기준, 팀 단위 도입 사례가 빠르게 늘면서 단순 개인 설정 도구에서 팀 컨벤션 자동화 인프라로 자리를 넓혀가고 있어요.

이 글에서는 실무 적용 후기를 바탕으로 코드 컨벤션 자동화가 팀 개발 효율에 어떤 변화를 만들었는지 살펴봐요.

> **핵심 요약**
> - Cursor Rules 파일은 프로젝트 루트에 두는 `.cursorrules` 또는 `.cursor/rules/*.mdc` 파일로, AI가 코드를 생성하거나 리뷰할 때 팀 컨벤션을 자동으로 적용하도록 만드는 설정 레이어예요.
> - W컨셉 프론트엔드 팀은 Cursor AI 기반 코드 리뷰 자동화 도입 후 반복 피드백 항목이 실질적으로 줄고, 리뷰 사이클 속도가 개선됐다고 Medium 게시물에서 밝혔어요.
> - 2025년 Cursor 0.43 버전부터 `.mdc` 형식의 MDC(Markdown with Context) 규칙이 공식 지원되면서, 규칙 범위를 훨씬 세밀하게 잡을 수 있게 됐어요.
> - 팀 단위 도입에서 가장 큰 병목은 기술 설정이 아니라 "무엇을 규칙으로 만들지" 합의하는 과정이에요.

## Cursor Rules 파일, 왜 지금 주목받나

Cursor는 원래 개인 개발자용 AI 코딩 도구로 알려져 있었어요. `.cursorrules` 파일도 처음엔 "나만의 AI 설정"이라는 느낌이 강했죠. 그런데 2025년 들어 팀 단위 채택 패턴이 눈에 띄게 달라졌어요.

배경엔 두 가지 변화가 있어요.

첫째, **MDC 형식 도입**이에요. Cursor 0.43 이후 `.cursor/rules/` 디렉토리 안에 `.mdc` 파일을 여러 개 놓고, 각 파일에 `glob` 패턴으로 적용 범위를 지정할 수 있게 됐어요. `components/**/*.tsx`에만 React 컨벤션 규칙을 적용하고, `api/**/*.ts`엔 별도 규칙을 얹는 식이죠. 기존 단일 파일로는 불가능했던 세분화예요.

둘째, **ESLint만으로는 부족했던 영역**이에요. ESLint는 문법 오류나 정해진 패턴 위반을 잡아요. 그런데 "이 팀은 에러 처리 시 `try/catch` 대신 Result 패턴을 써요"라든가 "컴포넌트 props는 interface보다 type으로 선언해요" 같은 맥락적 컨벤션은 린터로 잡기 어려워요. Cursor Rules는 이 틈새를 메워요. AI가 코드를 생성할 때 해당 맥락을 참고해서, 팀 스타일에 맞는 코드를 처음부터 내놓도록 유도하죠.

국내에서도 W컨셉 같은 이커머스 프론트엔드 팀이 실제 사례를 공개했고, JackerLab처럼 MDC 고급 기법을 정리한 기술 블로그도 늘고 있어요. "도입을 고민 중"에서 "어떻게 잘 쓸까"로 질문이 바뀐 셈이에요.

## 실무 적용에서 실제로 무엇이 달라지나

### 코드 리뷰 반복 피드백 감소

W컨셉 프론트엔드 팀의 적용기에서 가장 인상적인 부분은 리뷰 코멘트의 성격이 바뀌었다는 거예요. 도입 전엔 "네이밍", "import 순서", "타입 선언 방식" 같은 스타일 피드백이 코멘트의 상당 부분을 차지했어요. 도입 후엔 이런 항목이 AI 코드 생성 단계에서 이미 처리되니까, 리뷰어가 아키텍처나 로직 수준의 피드백에 집중할 수 있게 됐죠.

단순히 편해진 게 아니에요. 리뷰 사이클 자체가 짧아지고, 스타일 체크에 쓰던 인지 에너지를 더 중요한 곳에 쓸 수 있어요.

### 신규 팀원 온보딩 속도

여러 팀이 공통으로 언급하는 게 있어요. 신규 입사자가 팀 컨벤션을 "외우지 않아도" 코드를 맞게 쓸 수 있게 됐다는 거예요. Rules 파일이 살아있는 컨벤션 문서 역할을 하거든요. 기존엔 컨벤션 문서가 Notion에 있었는데, 실제 코딩할 때 그걸 열어보는 사람은 드물잖아요. Cursor Rules는 IDE 안에서 직접 작동하니까 문서와 실행 사이의 간극이 없어요.

### 규칙 관리의 현실적 비용

장밋빛만은 아니에요. Rules 파일도 코드처럼 유지보수가 필요해요. 라이브러리가 업데이트되거나 팀 컨벤션이 바뀌면 Rules 파일도 같이 갱신해야 해요. 이걸 안 하면 AI가 오래된 패턴을 계속 내놓아요. JackerLab의 MDC 가이드는 이 문제를 "Rules 파일의 기술 부채"라고 부르는데, 딱 맞는 표현이에요.

## `.cursorrules` vs `.cursor/rules/*.mdc`: 어떤 걸 써야 하나

| 항목 | `.cursorrules` (단일 파일) | `.cursor/rules/*.mdc` (MDC 방식) |
|------|--------------------------|----------------------------------|
| 설정 범위 | 프로젝트 전체 일괄 적용 | 파일 패턴별 세분화 적용 가능 |
| 규칙 개수 | 하나의 파일에 전부 | 여러 파일로 역할 분리 |
| 조건부 적용 | 불가 | `glob` 패턴으로 가능 |
| 팀 협업 | 단순하지만 충돌 가능성 | 파일 분리로 충돌 줄어듦 |
| 학습 비용 | 낮음 | 중간 (MDC 문법 이해 필요) |
| Cursor 지원 버전 | 전 버전 | 0.43 이상 |
| 적합한 팀 규모 | 1-3인, 소규모 프로젝트 | 4인 이상, 멀티 도메인 코드베이스 |

소규모 팀이거나 단순 프로젝트라면 `.cursorrules` 하나로도 충분해요. 프론트엔드, 백엔드, 공통 유틸리티가 섞인 코드베이스라면 MDC 방식이 훨씬 깔끔하게 관리돼요.

MDC 방식에서 한 가지 더. `alwaysApply: true` 옵션을 쓰면 특정 규칙이 모든 파일에 항상 적용돼요. 팀 전체 공통 컨벤션은 여기에 두고, 도메인별 규칙은 `glob`으로 좁히는 방식이 실무에서 가장 잘 돌아가는 패턴이에요.

## 팀 도입 전에 반드시 거쳐야 할 단계

Cursor Rules 도입의 핵심 병목은 설정 기술이 아니에요. "우리 팀의 컨벤션이 글로 정리되어 있는가?"가 먼저예요.

Rules 파일을 쓰려면 팀 컨벤션이 명문화돼 있어야 해요. 암묵적으로만 공유되던 규칙들, "우리 팀은 useEffect 남발 안 해요" 같은 것들을 텍스트로 꺼내는 작업이 선행돼야 하거든요.

**스타일 규칙부터 시작하는 팀**: ESLint가 잡지 못하는 타입 선언 방식, import 그루핑 규칙을 먼저 Rules에 담으세요. 효과가 빠르고 팀 피드백을 얻기 좋아요.

**아키텍처 패턴이 있는 팀**: "이 팀은 데이터 패칭에 React Query를 쓰고, SWR은 안 씁니다"처럼 선택한 패턴을 명시적으로 적으세요. AI가 엉뚱한 라이브러리를 제안하는 상황을 줄여줘요.

**Rules 파일 관리 주체를 정하지 않은 팀**: 도입 후 3개월이 지나도 Rules가 처음 그대로라면 효과가 떨어져요. 누가 언제 업데이트할지 PR 프로세스에 포함시키는 게 좋아요.

참고로, Cursor 팀이 2026년 중 Rules 파일의 버전 관리 및 팀 공유 기능 강화를 로드맵에서 언급하고 있어요. 현재 Git으로 직접 관리하는 방식에서 IDE 레벨의 팀 동기화로 발전할 가능성이 높아요.

## 정리: 코드 컨벤션 자동화, 다음 단계는

실무 적용 후기들이 공통으로 보여주는 건 세 가지예요.

- 스타일 레벨 리뷰 코멘트가 줄고, 리뷰의 질이 높아져요
- 신규 팀원 온보딩 시 컨벤션 학습 비용이 줄어요
- Rules 파일 자체가 팀 컨벤션 문서 역할을 해요

앞으로 6-12개월 안에 주목할 흐름은 **Rules 파일의 표준화**예요. 지금은 팀마다 포맷이 달라요. Cursor가 공식 Rules 스펙을 강화하면, Next.js, NestJS, FastAPI 같은 프레임워크별 "starter rules"가 GitHub에 공유되는 생태계가 만들어질 가능성이 높아요.

팀에서 Cursor를 이미 쓰고 있다면, 지금 당장 `.cursorrules` 파일 하나만 만들어 보는 게 가장 빠른 시작이에요. 처음엔 다섯 줄짜리 규칙으로 시작해도 충분해요. 규칙이 쌓이면서 팀 컨벤션이 어디에 있는지 명확해지거든요.

그게 결국 이 도구의 진짜 가치예요. AI를 잘 쓰는 것보다, **팀이 무엇에 동의하는지 명확히 하는 과정** 자체가 개발 효율을 바꿔요.
